## Coding Style & Naming Conventions

### General Rules
- **Indentation**: Use 2 spaces (no tabs)
- **Line Length**: Maximum 100 characters
- **Quotes**: Use single quotes for JavaScript, double quotes for HTML attributes
- **Semicolons**: Always use semicolons in JavaScript

### Naming Conventions

**Files:**
- Components: `PascalCase.js` (e.g., `UserProfile.js`)
- Utilities: `camelCase.js` (e.g., `formatDate.js`)
- Styles: `kebab-case.css` (e.g., `user-profile.css`)
- Tests: `[filename].test.js` (e.g., `UserProfile.test.js`)

**Code:**
- Variables & Functions: `camelCase` (e.g., `getUserData`)
- Classes & Components: `PascalCase` (e.g., `UserController`)
- Constants: `UPPER_SNAKE_CASE` (e.g., `API_BASE_URL`)
- CSS Classes: `kebab-case` (e.g., `user-profile-container`)

### Tools
- **Linter**: ESLint with project configuration
- **Formatter**: Prettier
- **Pre-commit Hooks**: Husky for automatic linting/formatting

## Testing Guidelines

### Testing Framework
- **Unit Tests**: Jest
- **Integration Tests**: Jest + Testing Library

## Commit & Pull Request Guidelines

### Commit Message Format
```
<type>(<scope>): <subject>

<body>

<footer>
```

**Types:**
- `feat`: New feature
- `fix`: Bug fix
- `docs`: Documentation changes
- `style`: Code style changes (formatting, no logic change)
- `refactor`: Code refactoring

**Examples:**

fix(api): resolve timeout issue in data fetching

docs(readme): update installation instructions

## Architecture Overview

### Design Patterns
- **Component-Based**: Modular, reusable components
- **Separation of Concerns**: Logic, presentation, and data layers are distinct
- **Service Layer**: External API calls isolated in service modules

### State Management
- Use context/state management library for global state
- Keep component state local when possible
- Avoid prop drilling beyond 2-3 levels

### Error Handling
- Always handle errors gracefully
- Provide user-friendly error messages
- Log errors for debugging purposes

## Security & Configuration Tips

### Environment Variables
- Never commit `.env` files
- Use `.env.example` to document required variables
- Store sensitive data in environment variables only

### Dependencies
- Regularly update dependencies for security patches
- Review dependency licenses before adding new packages
- Minimize external dependencies when possible

### Code Security
- Sanitize user inputs
- Avoid exposing sensitive information in client-side code
- Use HTTPS for all external API calls

## Agent-Specific Instructions for BAILUCODE AI IDE

When assisting with tasks in this repository:

1. **UI/Frontend Tasks**: 
   - Use FontAwesome icons or create SVG graphics (not emoji)
   - Ensure modern, professional component design
   - Separate CSS and JS into organized folders

2. **Complex Tasks**:
   - Create task lists in proper format
   - Mark tasks as complete progressively
   - Break down large features into manageable steps

3. **Project Setup**:
   - Review directory structure before starting
   - Create extensible, scalable folder organization
   - Never put all code in a single file

4. **Interactive Approach**:
   - Provide options for users to clarify requirements
   - Ask clarifying questions when scope is unclear
   - Confirm approach before major implementations
